## Introduction

## 简介

LangChain has been around for a little over a year and has changed a lot as it’s grown to become the default framework for building LLM applications. As we [previewed a month ago](https://blog.langchain.com/the-new-langchain-architecture-langchain-core-v0-1-langchain-community-and-a-path-to-langchain-v0-1/), we recently decided to make significant changes to the  LangChain package architecture in order to better organize the project and strengthen the foundation.

LangChain 面世已逾一年，在成长为构建大语言模型（LLM）应用的默认框架过程中经历了诸多变化。正如我们[一个月前预告](https://blog.langchain.com/the-new-langchain-architecture-langchain-core-v0-1-langchain-community-and-a-path-to-langchain-v0-1/)的那样，我们近期决定对 LangChain 包的整体架构进行重大调整，以更好地组织项目并夯实基础。

Specifically we made two large architectural changes: separating out [`langchain-core`](https://github.com/langchain-ai/langchain/tree/master/libs/core?ref=blog.langchain.com) and separating out partner packages (either into [`langchain-community`](https://github.com/langchain-ai/langchain/tree/master/libs/community?ref=blog.langchain.com) or [standalone partner packages](https://github.com/langchain-ai/langchain/tree/master/libs/partners?ref=blog.langchain.com)) from `langchain`.

具体而言，我们实施了两项重大的架构变更：一是将 [`langchain-core`](https://github.com/langchain-ai/langchain/tree/master/libs/core?ref=blog.langchain.com) 独立出来；二是将合作伙伴相关模块从 `langchain` 中拆分出去，分别归入 [`langchain-community`](https://github.com/langchain-ai/langchain/tree/master/libs/community?ref=blog.langchain.com) 或 [独立的合作伙伴包](https://github.com/langchain-ai/langchain/tree/master/libs/partners?ref=blog.langchain.com)。

As a reminder, `langchain-core` contains the main abstractions, interfaces, and core functionality. This code is stable and has been following a stricter versioning policy for a little over a month now.

需要提醒的是，`langchain-core` 包含核心抽象、接口及基础功能。这部分代码已趋于稳定，并在过去一个多月里严格遵循更严格的版本管理策略。

`langchain` itself, however, still remained on 0.0.x versions. Having all releases on minor version 0 created a few challenges:

然而，`langchain` 主包本身此前一直停留在 0.0.x 版本。所有发布版均处于次版本号为 0 的状态，由此带来若干挑战：

- Users couldn’t be confident that updating would not have breaking changes  
- `langchain` became bloated and unstable as we took a “maintain everything” approach to reduce breaking changes and deprecation notifications  

- 用户无法确信升级不会引入破坏性变更；  
- 为尽量减少破坏性变更与弃用提示，我们曾采取“维护全部内容”的策略，导致 `langchain` 日益臃肿且不稳定。

However, starting today with the release of `langchain` 0.1.0, all future releases will follow a new versioning standard. Specifically:

不过，自今日 `langchain` 0.1.0 发布起，所有后续版本将遵循全新的版本号规范。具体如下：

- Any breaking changes to the public API will result in a minor version bump (the second digit)  
- Any bug fixes or new features will result in a patch version bump (the third digit)  

- 对公共 API 的任何破坏性变更，将触发次版本号升级（即第二位数字）；  
- 任何缺陷修复或新增功能，将触发修订号升级（即第三位数字）。

We hope that this, combined with the previous architectural changes, will:

我们希望，这一举措，结合此前的架构调整，能够实现以下目标：

- Communicate clearly if breaking changes are made, allowing developers to update with confidence  
- 清晰地传达是否引入了不兼容变更，使开发者能够自信地完成升级；

- Give us an avenue for officially deprecating and deleting old code, reducing bloat  
- 为我们提供一条正式弃用并移除旧代码的路径，从而减少代码冗余；

- More responsibly deal with integrations (whose SDKs are often changing as rapidly as LangChain)  
- 更负责任地应对第三方集成（其 SDK 的更新节奏往往与 LangChain 一样快速）。

Even after we release a 0.2 version, we will commit to maintaining a branch of 0.1, but will only patch critical bug fixes. See more towards the end of this post on our plans for that.

即便在发布 0.2 版本之后，我们仍将承诺维护一个 0.1 版本的分支，但仅针对严重缺陷提供补丁修复。有关该计划的更多细节，请参见本文末尾部分。

While re-architecting the package towards a path to a stable 0.1 release, we took the opportunity to talk to hundreds of developers about why they use LangChain and what they love about it. This input guided our direction and focus. We also used it as an opportunity to bring parity to the Python and JavaScript versions in the core areas outlined below.

在将整个包重构、迈向稳定版 0.1 的过程中，我们借此机会与数百名开发者深入交流，了解他们为何选择 LangChain，以及他们最喜爱 LangChain 的哪些方面。这些宝贵反馈直接塑造了我们的战略方向与工作重心。同时，我们也借机推动 Python 与 JavaScript 两个版本在下文所列核心功能领域达成功能对等。

💡

💡

While certain integrations and more tangential chains may be language specific, core abstractions and key functionality are implemented equally in both the Python and JavaScript packages.

尽管某些集成模块及更边缘化的链式组件可能具有语言特异性，但核心抽象层与关键功能在 Python 和 JavaScript 两个版本中均得到了同等实现。

We want to share what we’ve heard and our plan to continually improve LangChain. We hope that sharing these learnings will increase transparency into our thinking and decisions, allowing others to better use, understand, and contribute to LangChain. After all, a huge part of LangChain is our community – both the user base and the 2000+ contributors – and we want everyone to come along for the journey.

我们希望分享所收集到的用户声音，以及我们持续改进 LangChain 的具体规划。我们相信，公开这些洞见将提升我们决策逻辑与思考过程的透明度，帮助大家更高效地使用、更深入地理解，并更积极地参与 LangChain 的共建。毕竟，LangChain 的一大核心正是我们的社区——既包括广大用户群体，也涵盖 2000 多位开源贡献者。我们诚挚邀请每一位成员，共同踏上这段旅程。

## Third Party Integrations

## 第三方集成

One of the things that people most love about LangChain is how easy we make it to get started building on _any_ stack. We have [almost 700 integrations](https://integrations.langchain.com/?ref=blog.langchain.com), ranging from LLMs to vector stores to tools for agents to use.

LangChain 最受用户喜爱的特点之一，便是它让开发者能极其便捷地基于 _任意_ 技术栈快速启动开发。目前，我们已提供 [近 700 种集成](https://integrations.langchain.com/?ref=blog.langchain.com)，覆盖大语言模型（LLM）、向量数据库，以及智能体（Agent）可调用的各类工具。

LangChain is often used as the “glue” to join all the different pieces you need to build an LLM app together, and so prioritizing a robust integration ecosystem is a priority for us.

LangChain 常被用作“粘合剂”，将构建大语言模型（LLM）应用所需的各种组件整合在一起；因此，打造一个稳健、可靠的集成生态系统，是我们工作的重中之重。

About a month ago, [we started making some changes](https://blog.langchain.com/the-new-langchain-architecture-langchain-core-v0-1-langchain-community-and-a-path-to-langchain-v0-1/) we think will improve the robustness, stability, scalability, and general developer experience around integrations. We split out ALL third party integrations into `langchain-community` – this allows us to centralize integration-specific work. We have also begun to split out individual integrations into their own packages. So far we have done this for [~10 packages](https://github.com/langchain-ai/langchain/tree/master/libs/partners?ref=blog.langchain.com), including OpenAI, Google and Mistral. One benefit of this is better dependency management - previously, all dependencies were optional, leading to some headaches when trying to install specific versions. Now if integrations are in their own package, we can more strictly version their requirements, leading to easier installation. Another benefit is versioning. Oftentimes, there are changes to the third party integrations, which require breaking changes. These can now be reflected on an individual integration basis with proper versioning in the standalone integration package.

大约一个月前，我们启动了一系列[架构调整](https://blog.langchain.com/the-new-langchain-architecture-langchain-core-v0-1-langchain-community-and-a-path-to-langchain-v0-1/)，旨在提升集成模块的鲁棒性、稳定性、可扩展性以及整体开发者体验。我们将**所有第三方集成**全部迁移至独立仓库 `langchain-community`——此举使我们能够集中管理与各集成相关的开发工作。此外，我们还开始将单个集成进一步拆分为各自独立的软件包。截至目前，已完成对包括 OpenAI、Google 和 Mistral 在内的约 [10 个软件包](https://github.com/langchain-ai/langchain/tree/master/libs/partners?ref=blog.langchain.com) 的拆分。这一调整带来两大优势：其一是更精细的依赖管理——过去所有依赖均为可选安装，导致在指定版本安装时常常引发兼容性问题；如今每个集成拥有专属包，我们便可对其依赖项实施更严格的版本控制，显著简化安装流程。其二是更灵活的版本管理——第三方集成常需进行不兼容升级（breaking changes），现在这些变更可针对单个集成独立发布，并通过其独立软件包中的语义化版本号清晰体现。

## Observability

## 可观测性（Observability）

Building LLM applications involves putting a non-deterministic component at the center of your system. These models can often output unexpected results, so having visibility into exactly what is happening in your system is integral.

构建 LLM 应用意味着将一个非确定性（non-deterministic）组件置于系统核心。这类模型往往会产生出人意料的输出，因此，全面掌握系统内部运行状况至关重要。

💡

We want to make `langchain` as observable and as debuggable as possible, whether through architectural decisions or tools we build on the side.

我们的目标是，无论通过架构设计还是配套工具，都将 `langchain` 打造成尽可能**可观测、可调试**的框架。

We’ve set about this in a few ways.

我们正从多个维度推进这一目标。

The main way we’ve tackled this is by building [LangSmith](https://docs.smith.langchain.com/?ref=blog.langchain.com). One of the main value props that LangSmith provides is a [best-in-class debugging experience](https://docs.smith.langchain.com/tracing?ref=blog.langchain.com) for your LLM application. We log exactly what steps are happening, what the inputs of each step are, what the outputs of each step are, how long each step takes, and more data. We display this in a user-friendly way, allowing you to identify which steps are taking the longest, enter a playground to debug unexpected LLM responses, track token usage and more. Even in private beta, the demand for LangSmith has been overwhelming, and we’re investing a lot in scalability so that we can release a public beta and then make it generally available in the coming months. We are also already supporting an [enterprise version](https://airtable.com/appwQzlErAS2qiP0L/shrGtGaVBVAz7NcV2?ref=blog.langchain.com), which comes with a within-VPC deployment for enterprises with strict data privacy policies.

其中最主要的方式是构建 [LangSmith](https://docs.smith.langchain.com/?ref=blog.langchain.com)。LangSmith 的核心价值之一，便是为您的 LLM 应用提供业界领先的[调试体验](https://docs.smith.langchain.com/tracing?ref=blog.langchain.com)：我们完整记录每一步执行过程、各步骤的输入与输出、每步耗时等关键数据，并以直观友好的界面呈现——您可快速定位性能瓶颈、进入交互式 Playground 调试异常的 LLM 输出、追踪 Token 使用量等。即便尚处于私有 Beta 阶段，LangSmith 已收获远超预期的用户需求；为此，我们正大力投入可扩展性建设，以期尽快推出公开 Beta 版，并在未来数月内实现全面开放。我们目前已同步支持[企业版](https://airtable.com/appwQzlErAS2qiP0L/shrGtGaVBVAz7NcV2?ref=blog.langchain.com)，该版本支持在客户自有 VPC 内部署，满足对数据隐私有严格合规要求的企业客户。

We’ve also tackled observability in other ways. We’ve long had built in [`verbose` and `debug` modes](https://python.langchain.com/docs/guides/debugging?ref=blog.langchain.com) for different levels of logging throughout the pipeline. We recently introduced methods to [visualize the chain you created](https://python.langchain.com/docs/expression_language/how_to/inspect?ref=blog.langchain.com#print-a-graph), as well as [get all prompts used](https://python.langchain.com/docs/expression_language/how_to/inspect?ref=blog.langchain.com#get-the-prompts).

我们在可观测性方面还采取了其他多项举措：长期以来，LangChain 原生支持 [`verbose` 和 `debug` 日志模式](https://python.langchain.com/docs/guides/debugging?ref=blog.langchain.com)，可在流水线各环节提供不同粒度的日志输出；近期，我们新增了[可视化您所构建链（Chain）结构图](https://python.langchain.com/docs/expression_language/how_to/inspect?ref=blog.langchain.com#print-a-graph)的功能，以及[一键获取所有已使用提示词（Prompts）](https://python.langchain.com/docs/expression_language/how_to/inspect?ref=blog.langchain.com#get-the-prompts)的能力。

## Composability

## 可组合性（Composability）

While it’s helpful to have prebuilt chains to get started, we very often see teams breaking outside of those architectures and wanting to customize their chain - not only customize the prompt, but also customize different parts of the orchestration.

虽然预构建的链有助于快速上手，但我们经常看到团队突破这些既有架构，希望深度定制自己的链——这不仅包括定制提示词（prompt），还涉及对编排流程中各个组件的灵活调整。

💡

💡

Over the past few months, we’ve invested heavily in [LangChain Expression Language](https://python.langchain.com/docs/expression_language/?ref=blog.langchain.com) (LCEL). This enables composition of arbitrary sequences, providing a lot of the same benefits as data orchestration tools do for data engineering pipelines ( [batching](https://python.langchain.com/docs/expression_language/interface?ref=blog.langchain.com#batch), [parallelization](https://python.langchain.com/docs/expression_language/how_to/map?ref=blog.langchain.com), [fallbacks](https://python.langchain.com/docs/expression_language/how_to/fallbacks?ref=blog.langchain.com)).

过去几个月，我们大力投入开发了 [LangChain 表达式语言](https://python.langchain.com/docs/expression_language/?ref=blog.langchain.com)（LCEL）。它支持任意序列的组合，为 LLM 应用带来了类似数据工程流水线中数据编排工具所具备的核心能力（如 [批处理](https://python.langchain.com/docs/expression_language/interface?ref=blog.langchain.com#batch)、[并行化](https://python.langchain.com/docs/expression_language/how_to/map?ref=blog.langchain.com)、[容错回退](https://python.langchain.com/docs/expression_language/how_to/fallbacks?ref=blog.langchain.com)）。

It also provides some benefits unique to LLM workloads - mainly LLM-specific observability (covered above), and streaming, covered later in this post.

它还提供了专为大语言模型（LLM）工作负载设计的独特优势——主要包括面向 LLM 的可观测性（前文已介绍）以及流式响应（streaming），后者将在本文后半部分详述。

The components for LCEL are in `langchain-core`. We’ve started to create higher level entry points for specific chains in LangChain. These will gradually replace pre-existing (now “Legacy”) chains, because chains built with LCEL will get streaming, ease of customization, observability, batching, retries out-of-the-box. Our goal is to make this transition seamless. Previously you may have done:

LCEL 的核心组件位于 `langchain-core` 包中。我们已在 LangChain 中为特定链型开始构建更高层级的入口函数。这些新接口将逐步取代现有（现称为“Legacy”/“传统”）链实现，因为基于 LCEL 构建的链天然支持流式响应、易于定制、可观测性、批处理及自动重试等能力。我们的目标是让这一迁移过程平滑无感。此前你可能这样使用：

```
ConversationalRetrievalChain.from_llm(llm, …)
```

We want to simply make it:

我们希望将其简化为：

```
create_conversational_retrieval_chain(llm, …)
```

Under the hood, it will create a specific LCEL chain and return it. If you want to modify the logic - no problem! Because it’s all written in LCEL it’s easy to modify part of it without having to subclass anything or override any methods.

其底层会创建一个特定的 LCEL 链并返回该对象。若你想修改逻辑？完全没问题！由于整个链均由 LCEL 编写，你可以轻松修改其中任一部分，而无需继承任何类或重写任何方法。

There are a lot of chains in LangChain, and a lot of them are heavily used. We will not deprecate the legacy version of the chain until an alternative constructor function exists and has been used and well-tested.

LangChain 中包含大量链型，其中许多已被广泛采用。在对应的新版构造函数正式发布、经过充分使用与严格测试之前，我们不会弃用任何传统链的旧版本。

## Streaming

LLMs 有时响应较慢。向终端用户表明系统正在处理请求（而非让用户面对空白屏幕干等）至关重要。这种“正在处理”的反馈，可以体现为：从大语言模型（LLM）流式输出 token，或流式输出中间步骤（尤其当调用的是较长运行时间的链（chain）或智能体（agent）时）。

我们在这两方面均投入了大量研发资源。所有基于 LCEL（LangChain Expression Language）构建的链均原生支持标准的 [`stream`](https://python.langchain.com/docs/expression_language/interface?ref=blog.langchain.com#stream) 和异步 [`astream`](https://python.langchain.com/docs/expression_language/interface?ref=blog.langchain.com#async-stream) 方法；此外，我们还做了大量优化工作，确保流式输出不仅限于 LLM 调用本身（例如，也支持在输出解析器 output parser 中实现流式处理）。所有链还统一提供标准的 [`astream_log`](https://python.langchain.com/docs/expression_language/interface?ref=blog.langchain.com#async-stream-intermediate-steps) 方法，用于流式输出整个 LCEL 链中每一步的执行日志。这些日志可进一步按需过滤，从而轻松提取关键中间步骤及其他相关信息。

💡

流式输出（包括 token 和中间步骤）是绝大多数 LLM 应用中至关重要的用户体验（UX）组件；而 LangChain 让你开箱即用地获得这一能力。

## Output Parsing

LangChain 的核心使用场景之一是“工具调用（Tool Usage）”——即利用大语言模型来调用其他外部工具。

💡

确保大语言模型以结构化格式返回信息，是支撑其在下游应用中执行具体操作的关键前提。

我们围绕这一需求投入大量精力，打造了出色的开发者体验，并提出了 [输出解析器（output parser）](https://python.langchain.com/docs/modules/model_io/output_parsers/?ref=blog.langchain.com) 这一核心概念。

One of the main ways to do this is with [OpenAI Function calling](https://python.langchain.com/docs/modules/model_io/output_parsers/types/openai_functions?ref=blog.langchain.com).  
实现这一目标的主要方式之一是使用 [OpenAI 函数调用（OpenAI Function calling）](https://python.langchain.com/docs/modules/model_io/output_parsers/types/openai_functions?ref=blog.langchain.com)。

We’ve made it easy to not only specify the output format (using Pydantic, JSON schema, or even a function) but also to easily work with the response.  
我们不仅简化了输出格式的指定（支持 Pydantic、JSON Schema，甚至原生函数），还大幅降低了响应结果的处理难度。

We also support several different encoding methods ( [JSON](https://python.langchain.com/docs/modules/model_io/output_parsers/types/json?ref=blog.langchain.com), [XML](https://python.langchain.com/docs/modules/model_io/output_parsers/types/xml?ref=blog.langchain.com), [Yaml](https://python.langchain.com/docs/modules/model_io/output_parsers/types/yaml?ref=blog.langchain.com)) for when you want to do this with a model that doesn’t support OpenAI Function calling and resort to using prompting.  
此外，当所用模型不支持 OpenAI 函数调用、需退而采用提示工程（prompting）时，我们也支持多种编码格式：[JSON](https://python.langchain.com/docs/modules/model_io/output_parsers/types/json?ref=blog.langchain.com)、[XML](https://python.langchain.com/docs/modules/model_io/output_parsers/types/xml?ref=blog.langchain.com) 和 [YAML](https://python.langchain.com/docs/modules/model_io/output_parsers/types/yaml?ref=blog.langchain.com)。

When you resort to using prompting, you also need proper instructions to tell the LLM how to respond – all output parsers come equipped with a `get_format_instructions` method to get those instructions.  
在采用提示工程时，还需向大语言模型（LLM）提供清晰、准确的格式化指令；为此，所有输出解析器（output parser）均内置 `get_format_instructions` 方法，可一键生成所需指令。

We’ve also invested in more advanced functionality around output parsers, like allowing them to stream through partial results as they are generated to improve user experience.  
我们还在输出解析器领域投入研发，实现了更高级的功能——例如支持流式传输（streaming）逐步生成的中间结果，从而显著提升用户体验。

This includes streaming partial results from structured formats like JSON, XML and CSV.  With output parsing, that can sometimes be tricky - in order to parse a JSON blob, most JSON parsers require a full JSON blob. A lot of our output parsers contain built-in logic to do this partial parsing.  
这包括对 JSON、XML 和 CSV 等结构化格式的中间结果进行流式输出。然而，结构化输出的流式解析往往颇具挑战性——大多数 JSON 解析器要求输入为完整、合法的 JSON 对象（blob）。为此，我们的许多输出解析器内建了智能逻辑，可稳健地完成“渐进式解析”（partial parsing）。

## Retrieval

## 检索（Retrieval）

One of the [main types of applications](https://python.langchain.com/docs/use_cases/question_answering/?ref=blog.langchain.com) we see developers building is applications that interact with their private data.  
开发者构建的[最主要应用类型之一](https://python.langchain.com/docs/use_cases/question_answering/?ref=blog.langchain.com)是能与私有数据交互的应用程序。

💡  

💡  

Being able to easily combine _your_ data with LLMs is an incredibly important part of LangChain.  
能否便捷地将 _您自己的数据_ 与大语言模型（LLMs）结合，是 LangChain 极其关键的核心能力。

This generally involves two different components - ingestion (preparing the data) and retrieval (retrieving the data), both of which we’ve built out.  
该能力通常涵盖两个核心环节：数据接入（ingestion，即数据准备）与数据检索（retrieval，即数据查询），LangChain 均已提供完整实现。

On the ingestion side, a big part of ingestion is splitting the text you are working with into chunks. While this may seem trivial, the best way to do so is often nuanced and often specific to the type of document you are working with.  
在数据接入侧，文本分块（chunking）是其中至关重要的一步。尽管看似简单，但最优分块策略往往高度依赖具体文档类型，且细节丰富、差异显著。

We have [15 different text splitters](https://python.langchain.com/docs/modules/data_connection/document_transformers/?ref=blog.langchain.com), some optimized for specific document types (like [HTML](https://python.langchain.com/docs/modules/data_connection/document_transformers/HTML_header_metadata?ref=blog.langchain.com) and [Markdown](https://python.langchain.com/docs/modules/data_connection/document_transformers/markdown_header_metadata?ref=blog.langchain.com)) to give developers maximal control over this process.  
我们提供了 [15 种不同的文本分块器（text splitters）](https://python.langchain.com/docs/modules/data_connection/document_transformers/?ref=blog.langchain.com)，其中部分专为特定文档类型优化（例如 [HTML](https://python.langchain.com/docs/modules/data_connection/document_transformers/HTML_header_metadata?ref=blog.langchain.com) 和 [Markdown](https://python.langchain.com/docs/modules/data_connection/document_transformers/markdown_header_metadata?ref=blog.langchain.com)），赋予开发者对该流程的极致控制力。

Relevant data often is changing though, and our ingestion system is designed for production, scaled applications.  
然而，相关数据往往处于持续更新状态；因此，我们的数据接入系统专为生产环境、规模化应用而设计。

We’ve exposed an [indexing API](https://python.langchain.com/docs/modules/data_connection/indexing?ref=blog.langchain.com) to allow you to re-ingest content while ignoring pieces that have NOT changed - saving on time and cost for large-volume workloads.  
我们开放了 [索引 API（indexing API）](https://python.langchain.com/docs/modules/data_connection/indexing?ref=blog.langchain.com)，支持仅对变更内容重新接入，自动跳过未改动部分——大幅节省大规模数据处理的时间与成本。

On the retrieval side, we’ve invested in more advanced methods while also making retrieval more production ready.  
在数据检索侧，我们既深耕前沿方法，也着力提升其生产就绪度（production readiness）。

We’ve implemented [advanced retrieval strategies](https://python.langchain.com/docs/modules/data_connection/retrievers/?ref=blog.langchain.com#advanced-retrieval-types) from academia (like FLARE and Hyde), created our own (like [Parent Document](https://python.langchain.com/docs/modules/data_connection/retrievers/parent_document_retriever?ref=blog.langchain.com) and [Self-Query](https://python.langchain.com/docs/modules/data_connection/retrievers/self_query?ref=blog.langchain.com)), and adapted some from other industry solutions (like [Multi-Query](https://python.langchain.com/docs/modules/data_connection/retrievers/MultiQueryRetriever?ref=blog.langchain.com) \- derived from query expansion commonly used in search).  
我们实现了来自学术界的[先进检索策略](https://python.langchain.com/docs/modules/data_connection/retrievers/?ref=blog.langchain.com#advanced-retrieval-types)（如 FLARE 和 Hyde），自主研发了新方法（如 [父文档检索器（Parent Document）](https://python.langchain.com/docs/modules/data_connection/retrievers/parent_document_retriever?ref=blog.langchain.com) 和 [自查询（Self-Query）](https://python.langchain.com/docs/modules/data_connection/retrievers/self_query?ref=blog.langchain.com)），并借鉴工业界方案进行了适配（如 [多查询（Multi-Query）](https://python.langchain.com/docs/modules/data_connection/retrievers/MultiQueryRetriever?ref=blog.langchain.com) —— 源于搜索领域常用的查询扩展技术）。

We’ve also made sure to support production concerns like [per-user retrieval](https://python.langchain.com/docs/use_cases/question_answering/per_user?ref=blog.langchain.com) \- crucial for any application where you are storing documents for multiple users together.  
我们亦充分考虑生产环境需求，例如支持 [按用户检索（per-user retrieval）](https://python.langchain.com/docs/use_cases/question_answering/per_user?ref=blog.langchain.com)——这对将多用户文档集中存储的任何应用都至关重要。

Importantly, while LangChain provides all the necessary components for building advanced retrieval systems, we are not overly opinionated on how to do so.  
重要的是，尽管 LangChain 提供了构建先进检索系统所需的全部组件，但我们并未强加某种“唯一正确”的实现范式。

This has led to many other libraries building on top of LangChain to provide a more opinionated approach to retrieval - like [EmbedChain](https://github.com/embedchain/embedchain?ref=blog.langchain.com) and [GPTResearcher](https://github.com/assafelovic/gpt-researcher?ref=blog.langchain.com).  
这种开放性催生了大量基于 LangChain 的上层库，它们以更明确的设计理念推进检索能力——例如 [EmbedChain](https://github.com/embedchain/embedchain?ref=blog.langchain.com) 和 [GPTResearcher](https://github.com/assafelovic/gpt-researcher?ref=blog.langchain.com)。

## Agents

## 代理（Agents）

One of the earliest things that LangChain became known for was agentic workloads. This can mean two things:

LangChain 最早广为人知的特性之一便是对“代理型工作负载”（agentic workloads）的支持。这主要包含两层含义：

1. Tool use: having an LLM call a function or tool  
1. 工具调用：让大语言模型（LLM）调用某个函数或工具  

2. Reasoning: how to best enable an LLM to call a tool multiple times, and in what order (or not call a tool at all!)  
2. 推理机制：如何最优地支持 LLM 多次调用工具，并决定调用顺序（甚至在某些情况下完全不调用任何工具！）

On the tool use side, we’ve largely covered the components we see as crucial:

在工具调用层面，我们已基本覆盖了我们认为至关重要的核心组件：

1. Integrations with a large number of third party tools  
1. 与大量第三方工具的集成  

2. Ways to structure the LLMs response to fit the input schema of those tools  
2. 对 LLM 输出进行结构化处理，使其严格匹配各工具所需的输入 Schema  

3. A flexible way to specify custom manners in which those tools should be invoked (LCEL)  
3. 提供灵活方式，自定义这些工具的具体调用逻辑（例如通过 LCEL 实现）

On the reasoning side, we have a [few different “Agent” methods](https://python.langchain.com/docs/modules/agents/agent_types/?ref=blog.langchain.com), which can largely be thought of as an LLM running in a loop, deciding each iteration which (if any) tool it needs to call and then observing the result of that tool. We incorporated [ReAct](https://python.langchain.com/docs/modules/agents/agent_types/react?ref=blog.langchain.com) (an early prompting strategy for doing so) from the beginning, and have quickly added in many other types, including ones that use [OpenAI Function Calling](https://python.langchain.com/docs/modules/agents/agent_types/openai_functions_agent?ref=blog.langchain.com), ones that use their [new tool-calling API](https://python.langchain.com/docs/modules/agents/agent_types/openai_tools?ref=blog.langchain.com), ones optimized for conversation, and more.

在推理机制层面，我们提供了[多种不同的“Agent”方法](https://python.langchain.com/docs/modules/agents/agent_types/?ref=blog.langchain.com)，其核心思想可大致理解为：让 LLM 在一个循环中持续运行，每一轮迭代自主判断是否需要调用工具、调用哪一个工具，并观察该工具返回的结果。我们自项目初期便集成了 [ReAct](https://python.langchain.com/docs/modules/agents/agent_types/react?ref=blog.langchain.com)（一种早期用于实现该目标的提示工程策略），并迅速扩展出更多类型，包括基于 [OpenAI Function Calling](https://python.langchain.com/docs/modules/agents/agent_types/openai_functions_agent?ref=blog.langchain.com) 的 Agent、基于其 [全新工具调用 API（tool-calling API）](https://python.langchain.com/docs/modules/agents/agent_types/openai_tools?ref=blog.langchain.com) 的 Agent、专为对话场景优化的 Agent，以及其他更多类型。

💡

💡

Through flexible and extensible tool support and advanced reasoning capabilities, LangChain has the become the default way to enable LLMs to take actions.

凭借灵活可扩展的工具支持与先进的推理能力，LangChain 已成为赋能大语言模型执行具体操作（take actions）的事实标准方案。

Similar to retrieval, while LangChain provides the building blocks for agents we've also seen several more opinionated frameworks built on top. A great example of this is [CrewAI](https://github.com/joaomdmoura/CrewAI?ref=blog.langchain.com), which builds on top of LangChain to provide an easier interface for multi-agent workloads.

与检索（retrieval）模块类似，尽管 LangChain 提供了构建 Agent 的基础组件，我们也看到多个更具观点性（opinionated）、更高层级的框架在其之上构建而成。其中典型代表是 [CrewAI](https://github.com/joaomdmoura/CrewAI?ref=blog.langchain.com)，它基于 LangChain 构建，为多智能体（multi-agent）工作负载提供了更简洁易用的接口。

## LangChain 0.2

## LangChain 0.2

Even though we just released LangChain 0.1, we’re already thinking about 0.2. Some things that are top of mind for us are:

尽管我们刚刚发布了 LangChain 0.1，但我们已经在规划 0.2 版本了。目前我们重点关注的事项包括：

- Rewriting legacy chains in LCEL (with better streaming and debugging support)  
- 将遗留的链（legacy chains）重写为 LCEL 形式（并增强流式输出与调试支持）

- Adding new types of chains  
- 新增多种类型的链（chains）

- Adding new types of agents  
- 新增多种类型的智能体（agents）

- Improving our production ingestion capabilities  
- 提升生产环境中的数据接入（ingestion）能力

- Removing old and unused functionality  
- 移除陈旧且不再使用的功能

Importantly, even though we are excited about removing some of the old and legacy code to make `langchain` slimmer and more focused, we also want to maintain support for people who are still using the old version. That is why we will maintain 0.1 as a stable branch (patching in critical bug fixes) for at least 3 months after 0.2 release. We plan to do this for every stable release from here on out.

尤为重要的是：尽管我们非常期待通过移除部分陈旧和遗留代码，使 `langchain` 更加轻量、聚焦，但我们同样致力于持续支持仍在使用旧版本的用户。因此，在 0.2 版本发布后，我们将至少维持 0.1 版本作为稳定分支（仅合入关键 Bug 修复），为期不少于 3 个月。今后每一个稳定版本，我们都将延续这一策略。

And if you've been wanting to get started contributing, there's never been a better time. We recently added [a good getting started issue](https://github.com/langchain-ai/langchain/issues/15664?ref=blog.langchain.com) on GitHub if you're looking for a place to start.

如果你一直想开始参与贡献，那么现在正是最佳时机！我们最近在 GitHub 上新增了一个非常适合新手入门的议题：[a good getting started issue](https://github.com/langchain-ai/langchain/issues/15664?ref=blog.langchain.com)，欢迎从这里着手。

## One More Thing

## 还有一件事

A large part of LangChain v0.1.0 is stability and focus on the core areas outlined above. Now that we've identified the areas people love about LangChain, we can work on adding more advanced and complete tooling there.

LangChain v0.1.0 的一大重点在于稳定性，以及对上述核心领域的聚焦。如今，我们已明确用户最喜爱 LangChain 的方向，接下来便可围绕这些方向，构建更先进、更完备的工具体系。

One of the main things people love about LangChain is it's support for agents. Most agents are largely defined as running an LLM in some sort of a loop. So far, the only way we've had to do that is with [AgentExecutor](https://python.langchain.com/docs/modules/agents/concepts?ref=blog.langchain.com#agentexecutor). We've added a lot of parameters and functionality to AgentExecutor, but its still just one way of running a loop.

用户最钟爱 LangChain 的特性之一，便是其对智能体（agents）的强力支持。目前绝大多数智能体本质上都可被定义为“以某种循环方式调用大语言模型（LLM）”。迄今为止，我们实现该循环的唯一方式是借助 [AgentExecutor](https://python.langchain.com/docs/modules/agents/concepts?ref=blog.langchain.com#agentexecutor)。虽然我们已为 AgentExecutor 添加了大量参数与功能，但它终究只是实现循环的一种方式。

💡

💡

We're excited to announce the release of [`langgraph`](https://github.com/langchain-ai/langgraph?ref=blog.langchain.com), a new library to allow for creating language agents as graphs.

我们非常激动地宣布：全新库 [`langgraph`](https://github.com/langchain-ai/langgraph?ref=blog.langchain.com) 正式发布！它支持以图（graph）的形式构建语言智能体。

This will allow users to create far more custom cyclical behavior. You can define explicit planning steps, explicit reflection steps, or easily hard code it so that a specific tool is always called first.

这将使用户能够构建高度定制化的循环行为。你可以明确定义规划步骤（planning steps）、反思步骤（reflection steps），也可以轻松硬编码，确保某个特定工具始终被优先调用。

It is inspired by [Pregel](https://research.google/pubs/pub37252/?ref=blog.langchain.com) and [Apache Beam](https://beam.apache.org/?ref=blog.langchain.com).  
它受 [Pregel](https://research.google.pubs/pub37252/?ref=blog.langchain.com) 和 [Apache Beam](https://beam.apache.org/?ref=blog.langchain.com) 的启发。

The current interface exposed is one inspired by [NetworkX](https://networkx.org/documentation/latest/?ref=blog.langchain.com), and looks something like:  
当前公开的接口则借鉴自 [NetworkX](https://networkx.org/documentation/latest/?ref=blog.langchain.com)，其用法大致如下：

```python
from langgraph.graph import END, Graph

workflow = Graph()

workflow.add_node("agent", agent)
workflow.add_node("tools", execute_tools)

workflow.set_entry_point("agent")

workflow.add_conditional_edges(
    "agent",
    should_continue,
    {
        "continue": "tools",
        "exit": END
    }
)

workflow.add_edge('tools', 'agent')

chain = workflow.compile()
```

We've been working on this for the past six months, beta-testing it with users. It currently powers [OpenGPTs](https://github.com/langchain-ai/opengpts?ref=blog.langchain.com). We'll be adding a lot more examples and documentation over the next few weeks - we're really excited about this!  
过去六个月中，我们一直在开发这一项目，并已面向用户开展 Beta 测试。目前，它正为 [OpenGPTs](https://github.com/langchain-ai/opengpts?ref=blog.langchain.com) 提供支持。接下来几周内，我们将陆续补充大量示例和文档——对此我们倍感振奋！

[Try it out here.](https://github.com/langchain-ai/langgraph?ref=blog.langchain.com)  
[立即试用](https://github.com/langchain-ai/langgraph?ref=blog.langchain.com)
