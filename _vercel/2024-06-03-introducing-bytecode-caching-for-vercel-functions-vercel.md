2024 年 6 月 3 日

我们最近为 Vercel Functions 推出了一个基于 Rust 的新[核心](https://vercel.com/blog/vercel-functions-are-now-faster-and-powered-by-rust)，以提升启动速度。

今天，我们宣布推出一项新的实验性功能，旨在进一步降低大型应用的启动延迟，使冷启动速度最高提升 **27%**。

## Introducing bytecode caching

## 引入字节码缓存

冷启动过程中最耗时的环节之一，便是加载并编译 JavaScript 源代码。在执行代码之前，JavaScript 需先被解析并编译为 _字节码_（bytecode），随后由 V8 虚拟机直接执行该字节码，或由 V8 的即时编译器（JIT）进一步编译为机器码。

![](images/introducing-bytecode-caching-for-vercel-functions-vercel/img_001.jpg)![](images/introducing-bytecode-caching-for-vercel-functions-vercel/img_002.jpg)

这一字节码转换过程必须在 JavaScript 文件首次执行时完成，但会引入额外延迟。

如果我们能将这一步骤缓存起来，并在后续的冷启动中复用，会怎样？

![之前：每次冷启动时，JavaScript 都需重新编译为字节码。](images/introducing-bytecode-caching-for-vercel-functions-vercel/img_003.jpg)![之前：每次冷启动时，JavaScript 都需重新编译为字节码。](images/introducing-bytecode-caching-for-vercel-functions-vercel/img_004.jpg)之前：每次冷启动时，JavaScript 都需重新编译为字节码。

这正是字节码缓存（bytecode caching）的工作原理：首次执行生成字节码缓存，之后的执行及冷启动则复用并持续优化该缓存。该机制可透明地跳过编译步骤，从而缩短冷启动耗时。

![After: the bytecode is loaded and executed immediately.](images/introducing-bytecode-caching-for-vercel-functions-vercel/img_005.jpg)![After: the bytecode is loaded and executed immediately.](images/introducing-bytecode-caching-for-vercel-functions-vercel/img_006.jpg)  
![之后：字节码被立即加载并执行。](images/introducing-bytecode-caching-for-vercel-functions-vercel/img_005.jpg)![之后：字节码被立即加载并执行。](images/introducing-bytecode-caching-for-vercel-functions-vercel/img_006.jpg)  

After: the bytecode is loaded and executed immediately.  
之后：字节码被立即加载并执行。

We initially tested with three different Next.js applications which each load a different amount of JavaScript. Each application would get a cold start every 15 minutes. We compared the startup duration before and after bytecode caching.  
我们最初使用三个不同的 Next.js 应用进行了测试，每个应用加载的 JavaScript 体积各不相同。每个应用每 15 分钟触发一次冷启动。我们对比了启用字节码缓存前后函数的启动耗时。

- The first application’s main page is `250 kB`. The average TTFB went from 873ms to 764ms ( **-12%**) and the billed duration from 330ms to 137ms ( **-58%**)  
- 第一个应用的首页大小为 `250 kB`。平均首字节时间（TTFB）从 873ms 降至 764ms（**下降 12%**），计费时长从 330ms 降至 137ms（**下降 58%**）

- The second application’s main page is `550 kB`. The average TTFB went from 1017ms to 869ms ( **-15%**) and the billed duration from 463ms to 214ms ( **-54%**)  
- 第二个应用的首页大小为 `550 kB`。平均首字节时间（TTFB）从 1017ms 降至 869ms（**下降 15%**），计费时长从 463ms 降至 214ms（**下降 54%**）

- The third application’s main page is `800 kB`. The average TTFB went from 1548ms to 1130ms ( **-27%**) and the billed duration from 866ms to 453ms ( **-48%**)  
- 第三个应用的首页大小为 `800 kB`。平均首字节时间（TTFB）从 1548ms 降至 1130ms（**下降 27%**），计费时长从 866ms 降至 453ms（**下降 48%**）

These cold start improvements continue to improve as applications grow in size. With bytecode caching, your function both start faster and have a lower billed duration.  
随着应用体积增大，冷启动性能的提升效果持续增强。借助字节码缓存，您的函数不仅启动更快，计费时长也显著降低。

## Technical details  
## 技术细节

The [`v8-compile-cache`](https://github.com/zertosh/v8-compile-cache) npm package is widely known in the ecosystem, but can’t easily be used with serverless platforms like Vercel. The file system is ephemeral and fresh during a cold start.  
[`v8-compile-cache`](https://github.com/zertosh/v8-compile-cache) 这一 npm 包在生态中广为人知，但难以直接用于 Vercel 等无服务器平台——因为冷启动期间文件系统是临时且全新的。

We developed our own bytecode caching implementation that overcomes this limitation and allows all subsequent cold starts to use the produced bytecode cache. This system can continuously improve the cache as more traffic is sent to the function.  
我们自主研发了一套字节码缓存实现，成功克服了这一限制，使得后续所有冷启动均可复用已生成的字节码缓存。该系统还能随着函数接收更多流量而持续优化缓存内容。

For example, assume you have two routes `/home` and `/blog` . Your framework lazy-loads those routes from two JavaScript chunks. When a user hits `/home` for the first time, the bytecode generated by this first chunk is cached and re-used for future cold starts. But when a user then hits `/blog` , it produces a separate bytecode cache (since this chunk was lazy-loaded).  
例如，假设您有两个路由 `/home` 和 `/blog`，框架会分别从两个 JavaScript 代码块中懒加载这两个路由。当用户首次访问 `/home` 时，该代码块生成的字节码将被缓存，并在后续冷启动中复用；而当用户随后访问 `/blog` 时，则会生成一个独立的字节码缓存（因为该代码块同样是懒加载的）。

Vercel Functions will intelligently merge together all bytecode chunks, regardless of when or where they were created. This results in faster cold starts as your application gets more traffic.

Vercel Functions 会智能地合并所有字节码区块（bytecode chunks），无论这些区块是在何时、何地生成的。随着应用流量增加，这一机制可显著缩短冷启动时间。

## Try bytecode caching

## 尝试字节码缓存

We’ve been using bytecode caching on our internal Vercel projects for the past month.

过去一个月，我们已在内部 Vercel 项目中启用了字节码缓存功能。

If you want to try experimental bytecode caching, you will need to use Node.js 20 and use a framework that compiles to CommonJS (for example, Next.js). We plan to use the new option [available in Node.js 22](https://nodejs.org/en/blog/release/v22.1.0#module-implement-node_compile_cache-for-automatic-on-disk-code-caching) to support ES Modules in the future.

如需尝试实验性的字节码缓存功能，您需使用 Node.js 20，并选用编译为目标为 CommonJS 的框架（例如 Next.js）。未来，我们将利用 Node.js 22 中新增的选项（[详见 Node.js 22 发布日志](https://nodejs.org/en/blog/release/v22.1.0#module-implement-node_compile_cache-for-automatic-on-disk-code-caching)）来支持 ES 模块。

You can opt-in by adding the following Environment Variable in your project's settings, then re-deploy your application: `USE_BYTECODE_CACHING=1`. This improve only applies to production deployments.

您可通过在项目设置中添加以下环境变量并重新部署应用来启用该功能：`USE_BYTECODE_CACHING=1`。此优化仅对生产环境部署生效。
