Jan 7, 2026

2026年1月7日

Last year we introduced the [v0 Composite Model Family](https://vercel.com/blog/v0-composite-model-family), and described how the v0 models operate inside a multi-step agentic pipeline. Three parts of that pipeline have had the greatest impact on reliability. These are the dynamic system prompt, a streaming manipulation layer that we call “LLM Suspense”, and a set of deterministic and model-driven autofixers that run after (or while!) the model finishes streaming its response.

去年，我们推出了 [v0 复合模型家族（v0 Composite Model Family）](https://vercel.com/blog/v0-composite-model-family)，并介绍了 v0 模型如何在多步骤的智能体（agentic）流水线中运行。该流水线中有三个组件对系统可靠性影响最大：动态系统提示（dynamic system prompt）、我们称之为“LLM 悬念（LLM Suspense）”的流式处理操控层，以及一组确定性高、由模型驱动的自动修复器（autofixers）——这些修复器会在模型完成响应流式输出后（甚至在流式输出过程中！）立即运行。

**What we optimize for**

**我们的优化目标**

The primary metric we optimize for is the percentage of successful generations. A successful generation is one that produces a working website in v0’s preview instead of an error or blank screen. But the problem is that LLMs running in isolation encounter various issues when generating code at scale.

我们首要优化的指标是“成功生成率”（successful generations 的占比）。所谓成功生成，是指在 v0 的预览界面中能渲染出一个可正常运行的网站，而非报错或显示空白页。但问题在于：大语言模型（LLM）若孤立运行，在大规模代码生成任务中会频繁遭遇各类问题。

In our experience, code generated by LLMs can have errors as often as 10% of the time. Our composite pipeline is able to detect and fix many of these errors in real time as the LLM streams the output. This can lead to a double-digit increase in success rates.

根据我们的实践经验，LLM 生成的代码出错率高达约 10%。而我们的复合流水线能够在 LLM 流式输出响应的过程中实时检测并修复其中大量错误，从而将成功生成率提升十余个百分点。

### Dynamic system prompt

### 动态系统提示

Your product’s moat cannot be your system prompt. However, that does not change the fact that the system prompt is your most powerful tool for steering the model.

你产品的护城河（moat）绝不能依赖于你的系统提示（system prompt）。然而，这丝毫不削弱系统提示作为引导模型行为最有力工具的地位。

For example, take [AI SDK](https://ai-sdk.dev/) usage. AI SDK ships major and minor releases regularly. Models often rely on outdated internal knowledge (their “training cutoff”), but we want v0 to use the latest version. This can lead to errors like using APIs from an older version of the SDK. These errors directly reduce our success rate.

例如，考虑 [AI SDK](https://ai-sdk.dev/) 的使用场景。AI SDK 会定期发布主版本（major）和次版本（minor）更新。但模型往往依赖其过时的内部知识（即“训练截止时间”），而我们希望 v0 始终调用最新版 SDK。这种不一致可能导致错误，例如误用旧版 SDK 中已废弃的 API 接口——此类错误将直接拉低我们的成功生成率。

Many agents rely on web search tools for ingesting new information. Web search is great (v0 uses it too), but it has its faults. You may get back old search results, like outdated blog posts and documentation. Further, many agents have a smaller model summarize the results of web search, which in turn becomes a bad game of telephone between the small model and parent model. The small model may hallucinate, misquote something, or omit important information.

许多智能体依赖网络搜索工具来获取新信息。网络搜索固然强大（v0 同样使用它），但也存在固有缺陷：你可能检索到陈旧的结果，例如过时的博客文章或文档。此外，不少智能体还会让一个较小的模型对网络搜索结果进行摘要——这无形中演变成一场“传话游戏”（telephone game）：小模型与主模型之间反复转述，极易失真：小模型可能产生幻觉（hallucinate）、曲解原文，或遗漏关键信息。

Instead of relying on web search, we detect AI-related intent using embeddings and keyword matching. When a message is tagged as AI-related and relevant to the AI SDK, we inject knowledge into the prompt describing the targeted version of the SDK. We keep this injection consistent to maximize prompt-cache hits and keep token usage low.

我们不依赖网络搜索，而是通过嵌入（embeddings）和关键词匹配来识别与 AI 相关的用户意图。当某条消息被标记为与 AI 相关、且与 AI SDK 相关时，我们会将关于目标 SDK 版本的知识注入提示词（prompt）中。我们保持该注入方式的一致性，以最大化提示缓存（prompt-cache）命中率，并尽可能降低 token 消耗。

In addition to text injection, we worked with the AI SDK team to provide examples in the v0 agent’s read-only filesystem. These are hand-curated directories with code samples designed for LLM consumption. When v0 decides to use the SDK, it can search these directories for relevant patterns such as image generation, routing, or integrating web search tools.

除文本注入外，我们还与 AI SDK 团队协作，在 v0 代理的只读文件系统中提供了示例代码。这些是人工精选的目录，其中包含专为大语言模型（LLM）理解而设计的代码示例。当 v0 决定使用该 SDK 时，它可在这些目录中搜索相关模式，例如图像生成、路由逻辑，或集成网络搜索工具等。

These dynamic system prompts are used for a variety of topics, including frontend frameworks and integrations.

这类动态系统提示词被应用于多种主题场景，包括前端框架及各类集成方案。

### LLM Suspense

### LLM 悬停（LLM Suspense）

LLM Suspense is a framework that manipulates text as it streams to the user. This includes actions like find-and-replace for cleaning up incorrect imports, but can become much more sophisticated.

LLM Suspense 是一个在文本流式传输至用户过程中实时处理文本的框架。其功能包括查找并替换错误导入语句等基础操作，但也可实现更为复杂的文本变换逻辑。

Two examples show the flexibility it provides:

两个示例展示了该框架所具备的灵活性：

A simple example is substituting long strings the LLM often refers to. For example, when a user uploads an attachment, we give v0 a blob storage URL. That URL can be very long (hundreds of characters), which can cost 10s of tokens and impact performance.

一个简单示例是对 LLM 经常引用的长字符串进行替换。例如，当用户上传附件时，我们会向 v0 提供一个 Blob 存储 URL；该 URL 可能非常长（达数百个字符），导致消耗数十个 token，进而影响推理性能。

Before we invoke the LLM, we replace the long URLs with shorter versions that get transformed into the proper URL after the LLM finishes its response. This means the LLM reads and writes fewer tokens, saving our users money and time.

在调用 LLM 之前，我们将长 URL 替换为简短的占位符；待 LLM 完成响应后，再将这些占位符还原为原始 URL。这意味着 LLM 在处理过程中读写更少的 token，从而为用户节省费用与时间。

In production, these simple rules handle variations in quoting, formatting, and mixed import blocks. Because this happens during streaming, the user never sees an intermediate incorrect state.

在生产环境中，这些简单规则可妥善应对引号形式、格式排版以及混合导入块等多种变体。由于所有替换均发生在流式响应过程中，用户永远不会看到任何中间的错误状态。

![](images/how-we-made-v0-an-effective-coding-agent-vercel/img_003.jpg)![](images/how-we-made-v0-an-effective-coding-agent-vercel/img_004.jpg)

Suspense can also handle more complex cases. By default, v0 uses the [lucide-react](https://lucide.dev/guide/packages/lucide-react) icon library. It updates weekly, adding and removing icons. This means the LLM will often reference icons that no longer exist or never existed.

Suspense 还可处理更复杂的场景。默认情况下，v0 使用 [lucide-react](https://lucide.dev/guide/packages/lucide-react) 图标库。该库每周更新一次，增删图标。这意味着大语言模型（LLM）经常引用那些已不存在或根本从未存在的图标。

To correct this deterministically, we:

为确定性地修正此类问题，我们采取以下步骤：

1. Embed every icon name in a vector database.

1. 将每个图标名称嵌入向量数据库中。

2. Analyze actual exports from lucide-react at runtime.

2. 在运行时分析 `lucide-react` 的实际导出内容。

3. Pass through the correct icon when available.

3. 当正确图标可用时，直接透传该图标。

4. When the icon does not exist, run an embedding search to find the closest match.

4. 当所请求图标不存在时，执行向量嵌入搜索，找出最接近的匹配项。

5. Rewrite the import during streaming.

5. 在流式响应过程中重写导入语句。

For example, a request for a "Vercel logo icon" might produce:

例如，当请求一个“Vercel 徽标图标”时，可能生成如下代码：

```typescript
import { VercelLogo } from ‘lucide-react’
```

```typescript
import { VercelLogo } from ‘lucide-react’
```

LLM Suspense will replace this with:

LLM Suspense 将将其替换为：

```typescript
import { Triangle as VercelLogo } from ‘lucide-react’
```

该过程在 100 毫秒内完成，且无需额外调用模型。

### Autofixers（自动修复器）

有时，某些问题无法通过系统提示词（system prompt）或 LLM Suspense 机制解决。这类问题通常涉及跨多个文件的修改，或需要分析[抽象语法树](https://en.wikipedia.org/wiki/Abstract_syntax_tree)（AST）。

针对此类情况，我们会在流式响应完成后收集所有报错，并将其交由 autofixers（自动修复器）处理。这些修复器包括确定性修复逻辑，以及一个轻量、快速、经过微调的小型模型——该模型基于海量真实代码生成样本训练而成。

部分 autofix 示例包括：

- `@tanstack/react-query` 中的 `useQuery` 和 `useMutation` 必须包裹在 `QueryClientProvider` 内。我们通过解析 AST 来检查是否已正确包裹，而 autofix 模型则负责判断应在何处插入该 Provider。

- 通过扫描生成的代码，确定性地更新 `package.json`，补全缺失的依赖项。

- 修复那些在 Suspense 转换过程中遗漏的常见 JSX 或 TypeScript 错误。

这些修复操作耗时低于 250 毫秒，且仅在必要时触发，从而在保障低延迟的同时显著提升系统可靠性。

## Putting it together

## 整合各组件

Combining the dynamic system prompt, LLM Suspense, and autofixers gives us a pipeline that produces stable, functioning generations at higher rates than a standalone model.

将动态系统提示（dynamic system prompt）、LLM Suspense 与自动修复器（autofixers）相结合，构建出一条生成流程——相比单一模型，该流程能以更高成功率产出稳定、可运行的代码。

Each part of the pipeline addresses a specific failure mode, and together they significantly increase the likelihood that users see a rendered website in v0 on the first attempt.

流程中的每个环节都专门应对某一类典型失败模式；协同工作时，它们显著提升了用户在 v0 中首次尝试即成功渲染出网站的概率。
