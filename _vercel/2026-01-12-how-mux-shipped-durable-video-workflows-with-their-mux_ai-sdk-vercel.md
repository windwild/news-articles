_We invited_ [_Dylan Jhaveri_](https://x.com/dylanjha) _from_ [_Mux_](https://www.mux.com/) _to share how they shipped durable workflows with their_ [_@mux/ai SDK_](https://github.com/muxinc/ai) _._

我们邀请了来自 [_Mux_](https://www.mux.com/) 的 [_Dylan Jhaveri_](https://x.com/dylanjha)，分享他们如何借助其开源的 [_@mux/ai SDK_](https://github.com/muxinc/ai) 构建并交付具备持久化执行能力（durable workflows）的 AI 工作流。

AI workflows have a frustrating habit of failing halfway through. Your content moderation check passes, you're generating video chapters, and then you hit a network timeout, a rate limit, or a random 500 from a provider having a bad day. Now you're stuck. Do you restart from scratch and pay for that moderation check again? Or do you write a bunch of state management code to remember where you left off?

AI 工作流常常令人沮丧地在中途失败：内容审核检查顺利通过，视频章节生成也已启动，却突然遭遇网络超时、速率限制，或某家服务提供商“状态不佳”而返回的随机 500 错误。此时你进退维谷：是彻底重头开始——再次为内容审核付费？还是自行编写大量状态管理代码，以记录上一次执行中断的位置？

This is where durable execution changes everything.

而“持久化执行”（durable execution）正是彻底改变这一困局的关键。

When we set out to build [@mux/ai](https://github.com/muxinc/ai), an open-source SDK to help our customers build AI features on top of Mux's video infrastructure, we faced a fundamental question: how do we ship durable workflows in a way that's easy for developers to adopt, without forcing them into complex infrastructure decisions?

当我们着手开发 [@mux/ai](https://github.com/muxinc/ai) ——一个开源 SDK，旨在帮助客户基于 Mux 的视频基础设施快速构建 AI 功能时，我们面临一个根本性问题：如何交付具备持久化执行能力的工作流，使其易于开发者采用，同时又无需迫使他们做出复杂的基础架构决策？

The answer was Vercel's [Workflow DevKit](https://useworkflow.dev/).

答案是 Vercel 推出的 [Workflow DevKit](https://useworkflow.dev/)。

一个典型的视频 AI 工作流可能如下所示：

## 为何 Workflow DevKit 是最合适的选择

在评估各类解决方案时，我们确立了明确的设计原则：

- **无强制基础设施依赖。** 您应能在任意 Node.js 环境中直接运行 `@mux/ai` 中的函数，就像调用一个普通 SDK 那样简单。

- **按需启用持久化能力。** 如果您需要持久化存储、可观测性及错误处理机制，只需轻松叠加即可，不应增加额外复杂度。

- **Familiar patterns.** No new DSLs, no YAML, no state machine definitions. Just JavaScript.

- **熟悉的开发模式。** 无需学习新的领域特定语言（DSL），无需编写 YAML，也无需定义状态机——只需使用 JavaScript。

[Workflow DevKit](https://useworkflow.dev/) delivers on all of these. The `"use workflow"` and `"use step"` directives let us mark functions in the SDK for durable execution without changing how you write code. If you're in a standard Node environment, those directives are ignored. It's a no-op. If you're running in a Workflow DevKit environment, they give you automatic retries, state persistence, and observability.

[Workflow DevKit](https://useworkflow.dev/) 完全满足以上所有要求。“`"use workflow"`”和“`"use step"`”指令允许我们在 SDK 中标记函数，使其支持持久化执行，且完全不改变你原有的编码方式。如果你运行在标准的 Node 环境中，这些指令会被直接忽略——即为无操作（no-op）；而当你运行在 Workflow DevKit 环境中时，它们将自动为你提供重试机制、状态持久化以及可观测性能力。

This is the key insight with Workflow DevKit: the same code works everywhere, but gains durability guarantees when deployed to the right environment. And we're supporting Workflow DevKit without taking on an explicit dependency. If you're using Workflow DevKit: great, you get [all the benefits that come with that](https://useworkflow.dev/docs/getting-started). If you're not: no worries, you can still use `@mux/ai` as a normal Node package.

这是 Workflow DevKit 的核心设计洞见：同一套代码可在任意环境运行，但一旦部署到适配的环境，便自动获得持久化保障。更重要的是，我们对 Workflow DevKit 的支持是**零显式依赖**（zero explicit dependency）——如果你正在使用 Workflow DevKit：太好了，你将自动获得其[全部优势](https://useworkflow.dev/docs/getting-started)；如果你尚未采用：也完全没问题，你仍可将 `@mux/ai` 当作一个普通的 Node 包正常使用。

All the other options we considered would have required either taking on a specific third-party dependency and tying ourselves to one option, or building a more complex API surface so developers could add wrapping functions around each discrete step. It wasn't impossible, but it would have required a much bigger lift and a whole set of new decisions.

我们曾评估过的其他方案，要么需引入某个特定第三方依赖、从而将自身绑定于单一技术选型；要么需设计更复杂的 API 表面（API surface），以便开发者为每个独立步骤手动添加包装函数。这些方案并非不可行，但会显著增加工程复杂度，并引发一整套全新的架构决策。

Each `"use step"` function runs in isolation. If the moderation API fails after you've already generated the summary and tags, the workflow resumes from where it left off. You don't lose the work you've already paid for. The execution is distributed across multiple serverless function invocations, so long-running AI operations never hit timeout limits.

每个 `"use step"` 函数均独立运行。若在您已完成摘要与标签生成后，内容审核（moderation）API 却发生故障，工作流将从中断处继续执行，您此前已付费完成的计算工作不会丢失。整个执行过程被分发至多个无服务器函数调用中，因此耗时较长的 AI 操作永远不会触及超时限制。

The `@mux/ai` SDK also [ships with primitives](https://github.com/muxinc/ai?tab=readme-ov-file#primitives), which are lower-level singular units of work like `fetchTranscriptForAsset` and `getStoryboardUrl`. These have been exported with `"use step"` directives, so you can pull them into your own workflows and treat them as discrete steps.

`@mux/ai` SDK 还[内置了基础原语（primitives）](https://github.com/muxinc/ai?tab=readme-ov-file#primitives)，即更底层、功能单一的操作单元，例如 `fetchTranscriptForAsset` 和 `getStoryboardUrl`。这些原语均已通过 `"use step"` 指令导出，因此您可将其直接引入自己的工作流中，并作为独立步骤使用。

## Deploy anywhere, scale on Vercel

## 随处部署，在 Vercel 上弹性伸缩

Workflow DevKit is designed with portability in mind through the concept of "Worlds." A World is where workflow state gets stored. Locally it's JSON files on disk. On Vercel it's managed for you. Or you can [self-host with Postgres, Redis, or build your own](https://useworkflow.dev/docs/deploying#what-are-worlds). The easiest path is to develop and test locally with the Local World, then deploy to Vercel where everything is provisioned automatically, including the observability dashboard. This is the default zero-configuration experience.

Workflow DevKit 借助“World（世界）”这一概念，从设计之初便兼顾可移植性。“World”即工作流状态的存储场所：本地开发时，它对应磁盘上的 JSON 文件；在 Vercel 上则由平台自动托管；您也可[使用 PostgreSQL、Redis 自行托管，或构建自定义实现](https://useworkflow.dev/docs/deploying#what-are-worlds)。最便捷的路径是：先使用 Local World 在本地开发与测试，再一键部署至 Vercel——此时包括可观测性仪表板在内的全部基础设施均由平台自动配置完成。这正是开箱即用、零配置的默认体验。

For teams deploying to Vercel, this means:

面向部署至 Vercel 的团队，这意味着：

- **Zero infrastructure configuration.** Vercel detects durable functions and handles provisioning for you.  
- **零基础设施配置。** Vercel 可自动识别持久化函数（durable functions），并为您完成全部资源供给与配置。

- **Built-in observability.** Traces, logs, and metrics for every workflow run. Debug by replaying execution or time-traveling through state.

- **内置可观测性。** 每次工作流运行均自动生成追踪（traces）、日志（logs）和指标（metrics）。支持通过重放执行过程或“时间旅行”式回溯状态进行调试。

- **Automatic scaling.** Whether you're processing ten videos or ten thousand, the platform adapts without manual intervention.

- **自动扩缩容。** 无论您处理的是十段视频还是上万段视频，平台均可无需人工干预地动态适配。

## What you can build

## 您可以构建什么

`@mux/ai` ships with [pre-built workflows for common video AI tasks](https://github.com/muxinc/ai?tab=readme-ov-file#available-pre-built-workflows):

`@mux/ai` 集成了 [面向常见视频 AI 任务的预构建工作流](https://github.com/muxinc/ai?tab=readme-ov-file#available-pre-built-workflows)：

- **Summaries and tags:** Automatically understand and categorize your video library

- **摘要与标签：** 自动理解并分类您的视频资料库

- **Chapter generation:** Create navigation points based on content structure

- **章节生成：** 基于内容结构自动生成导航锚点

- **Content moderation:** Flag problematic content before it reaches users

- **内容审核：** 在问题内容触达用户前即予以标记

- **Translation and dubbing:** Make videos accessible in multiple languages

- **翻译与配音：** 让视频支持多种语言，提升可访问性

- **Embeddings:** Generate embeddings for nearest neighbor search

- **嵌入向量（Embeddings）：** 生成嵌入向量，用于最近邻搜索

All workflows are model-agnostic. Use OpenAI, Anthropic, or Gemini depending on the task. Everything is open source under Apache 2.0, so you can fork, modify, and extend as needed.

所有工作流均与模型无关。您可根据具体任务选用 OpenAI、Anthropic 或 Gemini 等大模型。全部代码均以 Apache 2.0 开源协议发布，您可自由 Fork、修改与扩展。
